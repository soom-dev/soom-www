## (a) Repo location and CI setup

**Decision: Separate repo `soom-dev/soom-www`.**

Rationale:
- Different framework (Astro + Svelte vs Node CLI), different deployment target (Cloudflare Pages vs npm), different release cadence.
- The CLI monorepo's CI (Vitest + Playwright against Node runtime) has nothing in common with a static marketing site's CI.
- Keeps the CLI repo focused; landing page designers don't need CLI dev deps.

CI plan:
- **Cloudflare Pages GitHub integration** for auto-deploy on push to `main`. Preview deploys on PRs.
- **GitHub Actions** for lint + Vitest + Playwright visual regression tests on every PR. Separate from deploy.
- Build command: `bun run build` → output `dist/`.

## (c) Animation library choice

**Decision: Neither Motion One nor GSAP. Use native APIs + Svelte transitions.**

| | Motion One | GSAP + ScrollTrigger | Native (chosen) |
|---|---|---|---|
| Bundle | ~3.8 KB gz | ~23 KB gz | 0 KB |
| Scroll-linked | `scroll()` utility | ScrollTrigger plugin | IntersectionObserver + scroll listener |
| License | MIT | GSAP license (free for public sites) | N/A |
| Svelte compat | Good | Manual | Native |

Rationale:
- **100 KB total JS budget is tight.** Svelte runtime is ~4 KB gz. Zero-library animation maximizes headroom.
- The required animations are achievable with native APIs:
  - **Hero reveal:** IntersectionObserver triggers CSS class swap (greyscale → color + SVG stroke animation). Zero JS for the static state; Svelte island hydrates on scroll.
  - **Product tour camera-pan:** `scroll` event + `requestAnimationFrame` driving `transform: translate3d()` on each panel. ~40 lines of Svelte.
  - **Scroll reveals:** CSS `@keyframes` + IntersectionObserver adding `.visible` class. Svelte's `transition:` directives for island content.
  - **SVG edge drawing:** CSS `stroke-dasharray` + `stroke-dashoffset` animation. Native, hardware-accelerated.
  - **Node lifecycle:** CSS transitions on `opacity`, `filter` (drop-shadow), `fill`. Staggered via `transition-delay`.
- **Cross-browser:** IntersectionObserver is supported in all modern browsers. No polyfills needed.
- **Fallback for older browsers:** Animations simply don't play; content is fully visible (progressive enhancement by Astro's static-first approach).

If the camera-pan scroll-linking proves janky in testing, Motion One's `scroll()` is the fallback (~1.5 KB gz for just that utility). GSAP is overkill for this page.

## (d) Embedded /p/<id> previews

**Decision: Inline SVG diagrams with CSS/Svelte animation. No iframes. No dependency on the cloud player.**

Reasoning:
- The cloud player (task 005) is a parallel workstream and may not be merged when this landing page ships. Hard dependency = launch blocker.
- 5 iframe-embedded animations would blow the page weight budget (each iframe loads a full Hansoom HTML page: ~80-150 KB).
- IntersectionObserver-triggered inline SVGs are faster, lighter, and more controllable than iframes.

Implementation:
- Each of the 5 product tour diagrams is a **hand-crafted SVG** styled with Sapphire Nightfall tokens.
- SVGs are inlined in Svelte components (not `<img>` tags) so CSS animations can target internal elements.
- Each diagram has a CSS animation loop:
  1. Nodes transition through rest → active → completed states (opacity + shadow + fill)
  2. Edges draw progressively via `stroke-dashoffset` animation
  3. Flow particles (small circles) travel along edge paths via CSS `offset-path`
- Animation only starts when the panel enters viewport (IntersectionObserver).
- Animation pauses when the panel leaves viewport (performance).
- `prefers-reduced-motion: reduce` → skip all motion, show the diagram in its fully-revealed (all-completed) state.

The 5 diagram types:
1. **Simple flow:** A → B → C → D (linear sequence, demonstrates basic animation)
2. **Parallel branching:** Fork → A | B | C → Join (shows concurrent paths)
3. **AI agent loop:** Input → Parse → LLM → Decide → [Act | Respond] → Output (feedback loop)
4. **Microservices:** API Gateway → Auth → Service mesh → DB/Cache/Queue (architecture diagram)
5. **ML pipeline:** Ingest → Clean → Feature → Train → Eval → Deploy (with data flow)

Future migration: When the cloud player ships, the inline SVGs can be progressively replaced with `<iframe src="hansoom.dev/p/<id>" loading="lazy">` with the current SVGs as poster images. This is additive, not a rewrite.

## (e) Deployment via Cloudflare Pages

**Decision: Static output mode deployed to Cloudflare Pages via GitHub integration.**

Setup:
1. Create Cloudflare Pages project linked to `soom-dev/soom-www` repo.
2. Build settings:
   - Framework preset: Astro
   - Build command: `bun run build`
   - Build output directory: `dist`
   - Node.js version: 20 (for Astro build)
3. Custom domain: `hansoom.dev` (apex) + `www.hansoom.dev` (redirect).
4. DNS: CNAME `hansoom.dev` → `soom-www.pages.dev` (Cloudflare manages this since the domain is likely already on Cloudflare DNS).

No `@astrojs/cloudflare` adapter needed — static output deploys directly as a static site. No server-side rendering, no edge functions for the landing page itself.

Preview deploys: Every PR gets a `<hash>.soom-www.pages.dev` preview URL automatically. This is critical for visual review before merge.

Headers/redirects: Configured via `public/_headers` and `public/_redirects` files (Cloudflare Pages convention).

Cache: Static assets get immutable cache headers. HTML gets `cache-control: public, max-age=0, must-revalidate` for instant deploys.
